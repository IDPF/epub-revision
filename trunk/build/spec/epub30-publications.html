› 1.1. Purpose and Scope

This document, EPUB Publications 3.0, defines publication-level semantics and conformance requirements for EPUB3, the third major revision of EPUB, an interchange and delivery format based on XML and Web Standards.

This document is meant to be read and understood in concert with the other documents that make up EPUB3. The EPUB3 Overview [EPUB3Overview], which provides an informative overview of EPUB and a roadmap to the rest of the EPUB3 documents, should be read first.

› 1.2. Relationship to Other Specifications

This specification depends on other sibling specifications, as described in [EPUB3Overview]. Sibling specifications include EPUB Content Documents 3.0 [ContentDocs30], Open Container Format 3.0 [OCF3], and EPUB Media Overlays 3.0 [MediaOverlays30].

This specification defines constructs based on XML [XML], and by default all terminology used in this specification has the same meaning as its use in [XML]. This specification also relies on XML Namespaces [XMLNS], and in this specification the term "XML namespace" (or just "namespace") has the meaning defined in that specification.

EPUB Publications 3.0 is a revision to, and supersedes, Open Package Format 2.0.1 [OPF2].

› Chapter 2. Introduction

This section is informative

EPUB is an interchange and delivery format for digital publications, based on XML and Web Standards. An EPUB Publication can be thought of as a a reliable packaging of Web content that represents a digital book, magazine, or other type of publication, with a declarative means to navigate through the content, and the ability to be distributed reliably for online and offline consumption.

This specification defines the semantics and conformance requirements for EPUB at the publication level. It defines the formats of the Package and NCX documents, as well as rules for how these documents and other Publication resources are associated to create a conforming EPUB Publication. This specification also defines rules for how EPUB Publications are processed by conforming Reading Systems.

› Chapter 3. EPUB Publications

This section describes the high-level content and Reading System conformance requirements for EPUB Publications. The sections following this one describe more specifically the requirements for Package Documents, NCX Documents, and Core Media Types.

› 3.1. Publication Content Conformance

A EPUB Publication minimally consists of the following Publication Resources:

  • Exactly one Package Document, conformant to the requirements defined in “Package Document Content Conformance”

  • Exactly one NCX Document, conformant to the requirements defined in “NCX Document Content Conformance”

  • One or more EPUB Content Documents, conformant to the requirements defined for EPUB Content Documents in [ContentDocs30]

  • Zero or more EPUB Style Sheets, conformant to the content conformance requirements defined for EPUB Style Sheets in [ContentDocs30]

  • Zero or more Media Overlay Documents, conformant to the content conformance requirements defined in [MediaOverlays30]

  • Zero or more additional resources, with content conformance requirements inferred from their media type unless explicitly specified differently by this specification or one of its sibling specifications.

An EPUB Publication is instantiated as an EPUB File, i.e. in a ZIP container conformant with the requirements in [OCF3].

› Chapter 4. Package Documents

› 4.1. Introduction

This section is informative

The fundamental publication-level construct of an EPUB Publication is the Package Document. The Package Document provides central information about the publication as a whole, and about the individual resources that comprise the publication:

  • Publication metadata - mechanisms for including and/or referencing metadata applicable to the entire Publication and particular resources within it.

  • A Publication manifest, which identifies (via URI) and describes (via MIME type) the set of resources that collectively comprise the Publication.

  • A spine - an ordered sequence of id references to top-level resources in the manifest, from which all other resources in the set can be reached or utilized. The spine defines the default "reading order" of the publication.

  • Navigation - a publication-wide declarative navigation layer.

  • Fallback chains - an optional means for publications to define an ordered list of items that can be considered informational equivalents, that a Reading System can choose between in rendering content

  • Bindings - an optional means of associating script-based implementations with custom media types.

› 4.2. Package Document Content Conformance

A Package Document must be a well-formed XML Document [XML].
A Package Document must be encoded in UTF-8 or UTF-16 [Unicode].
A Package Document must be valid to the Package Document schema as defined in Appendix A, Package Document Schema, and in addition must conform to all content conformance constraints expressed in “Package Document Definition”.
The Package Document filename should use the file extension .opf. Any document fragments (included via external general entities [XML] should not use the .opf extension.

› 4.4.1.2. Metadata extensibility

This specification subsets the RDFa 1.1 ([RDFa11 Core]) profile and prefix definition mechanisms to simplify the inclusion and extension of metadata vocabularies for defining publications. These new features allow core metadata properties to be added unprefixed and additionally work to standardize the inclusion of metadata across the publishing spectrum by predefining prefixes for the most commonly-used vocabularies.

All publications must reference the default RDFa profile on the root package element using the profile attribute. Unlike its RDFa equivalent, the profile attribute must only contain the single canonical URI of the EPUB default profile.

The following example shows how the default RDFa profile is defined. 

                    
<package profile="http://www.idpf.org/TODO">
    …
</package>

The addition of the default RDFa profile allows all properties from the EPUB3 Package Metadata Vocabulary to be referenced without using prefixes. The profile also defines a set of prefixes for use adding other vocabulary properties to EPUB Publications (refer to the profile for more information).

If the RDFa profile does not define a prefix for a needed vocabulary, it does not imply that the vocabulary cannot be included for use. The prefix attribute can also be attached to the package element to define additional prefixes. New prefixes are defined by including whitespace-separated prefix/URI pairings.

The following example shows how to defines a new prefix for a vocabulary. 

                        
<package prefix="foaf: http://xmlns.com/foaf/spec/">
    <metadata>
        <meta property="dcterms:creator" id="auth">Samuel Johnson</meta>
        <meta property="foaf:title" about="#auth">Dr.</meta>
    </metadata>
</package>

It is not permitted to use the prefix attribute to redeclare a prefix that has already been defined in the default RDFa profile. The [XMLNS] method of defining namespace prefixes must similary not be used to define vocabulary prefixes.

› 4.4.2. The metadata Element

› 4.4.2.1. Overview

The metadata element encapsulates information about the publication as a whole, including metadata that applies to the publication package and to the abstract content contained within the package. Metadata is expressed through a combination of three required Dublin Core Metadata Element Set ([DCMES]) properties — title, identifier and language — coupled with optional properties from DCMI Metadata Terms ([DCTERMS]). These metadata terms can be further enhanced and extended by a set of optional properties also defined in this specification.

Properties from other RDF vocabularies may be added to the metadata section to enhance package and content metadata and to supplement required and optional properties, but Reading System support for them should not be assumed.

› 4.4.2.2. Definition
Element Name

metadata

Occurrence

Required first child of package.

Attributes

The metadata element has no attributes defined in this specification.

Content Model

In any order: dc:identifier [required], dc:title [required], dc:language [required], Meta Element [optional], link [optional]

› 4.4.2.3. Package metadata
› 4.4.2.3.1. Dublin Core Metadata Vocabulary Transition

This specification represents the beginning of a shift from traditional XML-based grammars to RDFa vocabularies for expressing metadata. Unlike XML grammars, which intrude into host grammars and require specific accommodations to work, RDFa is a more natural and fluid means of expressing metadata, working within existing attributes to a greater degree of semantic richness than would otherwise be possible.

The addition of RDFa support for the expression of metadata now allows properties from a wide variety of vocabularies to be easily incorporated for expressing and refining information about the package and its contents. RDFa properties also do not require the importing of new grammars, but allow properties to be expressed via meta elements. (For more information, refer to the Metadata Extensibility Mechanism.)

Although the requirement to include the core title, identifier and language [DCMES] elements has not changed with this release, their inclusion is focused on maintaining backwards compatibility during this period of transition, and their deprecation is expected in a future version. Content authors are instead encouraged to begin using the new properties from the Dublin Core Metadata Terms ([DCTERMS]) vocabulary, including the properties that supersede the required DCMES elements (see add link to override).

› 4.4.2.3.2. The DCMES identifier Element

Every metadata section must include at least one identifier element containing an unambiguous identifier for the package. Multiple identifier elements are permitted, but only one can be marked as the primary identifier for the package via the package element's unique-identifier attribute.

The following example shows the unique identifier element for a publication. 

                    
<package unique-identifier="pub-id">
    <metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
        <dc:identifier id="pub-id">9780101010101</dc:identifier>
    </metadata>
</package>

The [DCTERMS] identifier property will supersede the identifier element in a future version of the specification, but its inclusion is currently optional. Authors are encouraged to include both versions using the prefer attribute on the identifier element to indicate to Reading Systems that the property should be used in preference.

The following example shows how an identifier element and property should co-exist within a publication. 

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:identifier id="pub-id" prefer="isbn-id">9780101010101</dc:identifier>
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
</metadata>

This specification imposes no additional restrictions or requirements on identifiers except that they must be at least one character in length.

If the identifier (element or property) conforms to a established system or has been granted by an issuing authority, the scheme employed should be identified by attaching the optional scheme ancillary property to the identifier.

The following example shows how an identifier can be marked as an ISBN. 

                    
<metadata>
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
    <meta about="#isbn-id" property="scheme">isbn13"</meta>
</metadata>

This specification does not require or endorse the use of any scheme for identifiers, but authors are encouraged to use established identifiers for schemes such as provided by the ONIX and MARC standards. The scheme identifier used should conform to the requirements of the chosen system; this specification imposes no restrictions or requirements on scheme identifiers.

› 4.4.2.3.3. The DCMES title Element

Every metadata section must include at least one title element containing the title for the publication. Multiple title elements are permitted, but the optional sort-order property should be attached to each to indicate their primacy for display and other rendering purposes. If no means of establishing the primary title for the publication is included, Reading Systems must treat the first title element encountered as the primary title.

The following example shows how to indicate the display order for multiple title elements. 

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:title id="long-title">On the Origin of Species by Means of Natural Selection</dc:title>
        <meta property="sort-order" about="#long-title">2</meta>
    
    <dc:title id="short-title">On the Origin of Species</dc:title>
        <meta property="sort-order" about="#short-title">1</meta>
    
    <dc:title id="full-title">On the Origin of Species by Means of Natural Selection, 
        or the Preservation of Favoured Races in the Struggle for Life</dc:title>
        <meta property="sort-order" about="#full-title">3</meta>
</metadata>

The [DCTERMS] title property will supersede the title element in a future version of the specification, but its inclusion is currently optional. Authors are encouraged to include both versions using the prefer attribute on the title element to indicate to Reading Systems that the property should be used in preference.

The following example shows how a title element and property should co-exist within a publication. 

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:title id="short-title" prefer="dcterms-short-title">On the Origin of Species</dc:title>
        <meta property="dcterms:title" id="short-title">On the Origin of Species</meta>
</metadata>

This specification imposes no additional restrictions or requirements on titles except that they must be at least one character in length.

The file-as property can be attached to a title to include a normalized representation of the title for machine and bibliographic sorting purposes.

The following example shows a title and its normalized form for sorting. 

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:title id="pub-title">Les Misérables</dc:title>
        <meta property="file-as" about="#pub-title">Misérables, les</meta>
</metadata>

The alternate-script property can also be attached to provide a rendition of the title in another language and/or script.

The following example shows alternate renditions of a title for French and English readers. 

                    
<package xml:lang="fr">
    <metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
        <dc:title id="pub-title">Les Misérables</dc:title>
        <meta property="alternate-script" 
                    about="#pub-title" xml:lang="en">Les Miserables</meta>
    </metadata>
</package>

› 4.4.2.3.4. The DCMES language Element

Every metadata section must include a single language element containing the default language of the publication content. The language specified must be either:

  • a 2-letter language code as defined in [ISO 639-1]; or

  • a 3-letter language code as defined in [ISO 639-2/T].

The language identifier may include an optional dash-separated country code as defined in [ISO 3166].

The following example shows a publication in U.S. English.

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:language>en-US</dc:language>
</metadata>

The [DCTERMS] language property will supersede the language element in a future version of the specification, but its inclusion is currently optional. Authors are encouraged to include both versions using the prefer attribute on the language element to indicate to Reading Systems that the property should be used in preference.

The following example shows how a language element and property should co-exist within a publication. 

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:language prefer="pub-lang">ko</dc:language>
    <meta property="dcterms:language" id="pub-lang">ko</meta>
</metadata>

› 4.4.2.4. The meta element
› 4.4.2.4.1. Overview

The meta element provides a generic means of expressing metadata, both to express primary metadata about the package or content or to refine that metadata. All meta elements must attach the RDFa property attribute ([RDFa11 Core]) that contains a CURIE that resolves to a property from an RDF vocabulary (it is not required that Reading System be able to resolve the CURIES).

If a meta element only includes a property attribute, the metadata is assumed to apply to the package or content as a whole.

The following example shows an instance of package metadata.

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
</metadata>

To enahnce metadata properties defined in the [DCMES] elements or in other meta elements, the RDFa about attribute must be attached. The value of this attribute must be an IDREF to the xml:id of the element being refined.

The following example shows an instance of ancillary metadata.

                    
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
</metadata>
› 4.4.2.4.2. Definition
Element Name

meta

Occurrence

As child of the metadata element. Repeatable.

Attributes
property [required]

Specifies the by name a DCTERMS or ancillary metadata property being expressed.

about [optional unless required]

A reference by id to the subject of the meta element property being expressed. Optional only when a meta element contains a non-empty child text node. When the meta element contains an empty child text-node, the about attribute is required and must contain a valid id of another meta element.

Content Model

Text or Empty.

› 4.4.3.1. Overview

The manifest element provides an exhaustive list of all Publication Resources that constitute the Publication via its collection of child item elements.

All Publication Resources must be referenced in the manifest regardless of their physical location (i.e., regardless of whether they are available in the Container or remotely). Refer to “Publication Resource Locations” for media type specific requirements regarding resource locations.

This specification and its sibling specifications may explicitly define certain resource types to be external, thereby declaring that the resource is not a Publication Resource. Examples of such external resources include resources referenced from the Package Document metadata link element.

The manifest is not self-referencing: it must not include item elements referring to the resource(s) that make up the Package Document itself.

The order of item elements in the manifest is not significant. The presentation sequence of Content Documents is provided in spine .

› 4.4.3.3.3. Item Fallbacks

Unless one of the exceptions listed in “Fallback Requirements and Processing Flow ” apply, each item in the manifest whose media-type value is not one of the EPUB3 Core Media Types must use the fallback attribute to specify a fallback chain that includes at least one item whose media-type is an EPUB3 Core Media Type.

The fallback attribute IDREF [XML] value must resolve to another item in the manifest. The ordered list of all ID references that can be reached starting from a given item's fallback attribute is considered to be the "fallback chain" for that item. There must not be any circular- or self references in fallback chains.

The general rule for processing manifest-level fallbacks is that a Reading System that does not support the Media Type of a given item must traverse the fallback chain until it encounters a supported Media Type. The resource identified by the fallback item is then used in place of the unsupported item. It is legal to have a fallback attribute on an item of a Core Media Type and a Reading System may choose to "fall over" such references in order to find the optimal version of a resource to render in a given context.   • src - URI to the resource (value of the data attribute in the object element)

  • type - resource media type (value of the type attribute in the object element)

Any param children of the object element are encoded as URL parameters in the same manner; e.g. for: 

<object data="horse.ogg" media-type="audio/ogg"/>
<param name="autoplay" value="false">
</object>

a Reading System would instantiate the designated handler Content Document as if its URI query string parameters were: 

src=horse.ogg&type=audio/ogg&autoplay=false

Non-ASCII and non-URL-safe characters (e.g. '=') should be escaped in the usual manner for these parameters (UTF-8 encoding and using %XX escapes).

   <mediaType media-type="application/x-demo-slideshow" handler="impl"/>
 </bindings>

› Chapter 5. NCX Documents

› 5.1. Introduction

This section is informative

In order to increase efficacy and richness of navigation, EPUB Publications include a "Navigation Center eXtended," the NCX. The NCX provides a means for the author to provide a multi-purpose declarative global navigation layer that is independent of the content it references.

The NCX was originally standardized by the DAISY Consortium, as defined in Section 8 of [Z3986-2005] . The EPUB specification defines its own schema (XML grammar) for the NCX content type, as some multimedia and metadata components from the DAISY specification are not included.

› 5.1.1. Key NCX Components

The following are the key components of the NCX document:

navMap

The navMap exposes the hierarchical structure of the EPUB Publication. The provided structure is global, i.e. it transcends the boundaries of each individual Content Document ([ContentDocs30]) in the spine.

In terms of the number of links provided, the navMap is often more extensive than a traditional print table of contents.

navList

A navList is a container for non-hierarchical navigational information that is secondary to the navMap.

Multiple navLists can be provided, either representing constructs with homogenous semantics (such as list of illustrations, or all quotes of a certain author in an anthology) or with heterogenous semantics (such as the the landmarks navList).

pageList

The pageList is a special-purpose navList that allows representing the location of page boundaries present in a print source. Metadata for bibliographical identification of the print source can be defined via the navMeta element (and its meta children).

› 5.1.2. NCX Metadata

The NCX provides two mechanisms for expressing metadata:

Document metadata

The NCX head element allows the expression of metadata concerning the NCX Document as a whole. This version of this specification does not require head metadata to be provided.

Link target metadata

The epub:type attribute, as occurring on the navList, navPoint and navTarget elements, provides machine-readable metadata about the structural semantics of link target(s). Converserly, the navLabel element provides human-readable information.

Additionally, the navMeta element can be used to define metadata information (using the meta element) for the navList, navMap and pageList containers, as well as for the navPoint, navTarget and pageTarget elements.

Mappings to metadata vocabularies used when expressing document and link target metadata are created using the profile and prefix attributes on the ncx root element . The EPUB Structural Semantics Vocabulary is the default metadata vocabulary of the NCX.

› 5.2. NCX Document Content Conformance

An NCX Document must be a well-formed XML Document [XML].
An NCX Document must be encoded in UTF-8 or UTF-16 [Unicode].
An NCX Document must be valid to the NCX Document schema as defined in Appendix B, NCX Document Schema, and in addition must conform to all content conformance constraints expressed in “NCX Document Definition”.
The NCX Document filename should use the file extension .ncx. Any document fragments (included via external general entities [XML] should not use the .ncx extension.

› 5.3. NCX Reading System Conformance

EPUB Reading Systems must provide means to reach parts of a publication based on the information available within the NCX, by exposing the hierarchical navigation structure and the navigation lists specified by navMap, navList and pageList.
EPUB Reading Systems must use human-readable descriptions encoded within the navLabel element as authoritative information for rendering the hierarchical navigation structure and the navigation lists.

› 5.4.5.1. The pageTarget Element

The NCX pageTarget element specifies a pointer to content and briefly describes the target.

When a page break is present in the targeted Content Document ([ContentDocs30]) (for example, an element with the epub:type="pagebreak" attribute), and when a page number is associated with this page break, then the text element in the navLabel of a pageTarget must contain the textual representation of the page number specified in the EPUB Content Document. Note that there is no requirement for page numbers to be represented by numerical values, although of course this is a common authoring practice.

› 5.4.6.1. The landmarks navList

The landmarks navList identifies fundamental structural components of the publication, in order to enable Reading Systems to provide the user effective access to them.

The landmarks navList is identified through the landmarks value of the navList epub:type attribute. The landmarks navList is optional, and must not occur more than once within an NCX Document instance.

› 5.4.9. The navLabel Element

The NCX navLabel element briefly describes the target in the Content Document ([ContentDocs30]) . When applied to navPoints, it generally contains the heading text of the referenced section of the document.

There can be multiple navLabel elements, for example to provide information in various languages. Several navLabel elements with the same language (see xml:lang definition below) are forbidden. Since a navLabel element without the optional xml:lang attribute inherits the language specified for the ncx top-level element, only one navLabel element without the xml:lang attribute is allowed. Reading systems should first attempt to find a navLabel matching the language specified by user preferences, secondly matching the language specified by the package's dc:language metadata element, thirdly matching the language specified by the xml:lang attribute on the ncx element, and lastly by defaulting to the first available navLabel element (in XML document order).

› 6.2. Fallback Requirements and Processing Flow

This section defines in detail the requirements for including fallback content in a EPUB Publication, and the process by which Reading Systems must process fallback information. The rules for fallback processing depend on both the type of item, and how is referenced by a Content Document or other resource.

Unless specified otherwise in the sections below, fallbacks must be provided for any Publication Resource that is not a Core Media Type using the manifest fallback mechanism.

› 6.2.1. Audio and Video

A Reading System that supports video playback must support video files that utilize the MPEG-4 container with H.264/MPEG-4 AVC baseline profile format (H.264) video codec and MPEG-4 Advanced Audio Coding - Low Complexity (AAC-LC) audio codec, and this format is a Core Media Type.

A Reading System that supports audio playback must support MPEG-1 Layer 3 (MP3) audio, and this format is a Core Media Type. Any Reading System that supports video playback must also support standalone AAC-LC audio playback, but AAC-LC is NOT a Core Media Type.

› 6.2.2. XHTML Content Documents

For every Publication Resource referenced from an XHTML Content Document that is not a Core Media Type, suitable fallback(s) to a resource that is a Core Media Type must be provided as detailed below:

object

For resources referenced via the object element, legitimate fallback(s) must be provided via that element’s intrinsic replacement capabilities.

Reading Systems must ignore manifest-level fallbacks in resolving such references. (Note that the bindings element could be considered to enable a type of fallback, but not in the sense meant here of ensuring that content can be rendered on all Reading Systems, whether or not they support scripting).

img

For resources referenced via img elements, the text value of the alt attribute provides a legitimate fallback and should be included. Additional manifest-level fallbacks may be provided.

Reading Systems should process manifest-level fallbacks in resolving such references.

All other cases

For all other resource referencing situations in XHTML content documents, a legitimate fallback must be provided via the manifest fallback mechanism.

Reading Systems must process manifest-level fallbacks in resolving such resource references.

Only XHTML Content Documents must be referenced via iframe elements; fallback processing is inapplicable.

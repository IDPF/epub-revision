› Overview

› Purpose and Scope

This specification, EPUB Publications 3.0, defines publication-level semantics and conformance requirements for EPUB3, the third major revision of EPUB, an interchange and delivery format based on XML and Web Standards.

This specification is one of a family of related specifications that compose EPUB3, the third major revision of an interchange and delivery format for digital publications based on XML and Web Standards. It is meant to be read and understood in concert with the other specifications that make up EPUB3:

  • The EPUB3 Overview [EPUB3Overview], which should be read first, provides an informative overview of EPUB and a roadmap to the rest of the EPUB3 documents.

  • EPUB Content Documents 3.0 [ContentDocs30], which defines profiles of XHTML, SVG and CSS for use in the context of EPUB Publications.

  • EPUB Open Container Format (OCF) 3.0 [OCF3], which defines a file format and processing model for encapsulating a set of related resources into a single-file (ZIP) EPUB Container.

  • EPUB Media Overlays 3.0 [MediaOverlays30], which defines specifies a usage of SMIL, the Package Document, the EPUB Style Sheet and the EPUB Content Document for representation of synchronized text and audio publications.

This specification supersedes Open Package Format 2.0.1 [OPF2]. Refer to [CHANGES] for information on differences between this specificaton and its predecessor.

› Terminology

Core Media Type

A set of Publication Resource types for which no fallback is required. Refer to Core Media Types for more information.

EPUB Container (or Container)

A ZIP-based packaging and distribution format for an EPUB Publication, as defined in OCF Physical Container: The ZIP Container [OCF3].

EPUB Content Document (or Content Document)

A content document included in the spine that conforms to the constraints of one of the core EPUB Content Document vocabularies (XHTML and SVG).

An EPUB Content Document is a Core Media Type, and may therefore be included in the EPUB Publication without the provision of fallbacks .

EPUB Navigation Document

A a specialization of the XHTML Content Document, containing human- and machine readable global navigation information, conforming to the constraints expressed in EPUB Navigation Documents [ContentDocs30].

EPUB Publication (or Publication)

A logical document entity consisting of a set of interrelated resources, as defined by this specification and its sibling specifications.

EPUB Reading System (or Reading System)

A system that processes EPUB Publications for presentation to end users in a manner conformant with this specification and its sibling specifications.

EPUB Style Sheet (or Style Sheet)

A CSS Style Sheet conforming to the CSS profile defined in EPUB Style Sheets [ContentDocs30].

External Resource

A resource that is referenced from within an EPUB Publication, but is not a Publication Resource.

Foreign Content Document

A content document included in the spine that does not a conformant EPUB Content Document. A Foreign Content Document is not a Core Media Type, and the EPUB Publication must therefore include fallbacks for it, as defined in Fallback Requirements and Processing Flow .

Manifest

An list of all Publication Resources that constitute the EPUB Publication.

Refer to manifest for more information.

› Introduction

This section is informative

Where EPUB represents as an interchange and delivery format for digital publications based on XML and Web Standards, an EPUB Publication represents a specific instance of packaged Web content conforming to the specification, such as a digital book, magazine, or other type of publication. A Publication is more than just content, however; it also provides a declarative means of navigating through the publication plus the ability to be distributed reliably for online and offline consumption.

This specification defines the semantics and conformance requirements for EPUB at the publication level. It defines the format of the Package Document, as well as rules for how this document and other Publication Resources are associated to create a conforming EPUB Publication. This specification also defines conformance criteria for EPUB Reading Systems.

› Package Documents

› Introduction

This section is informative

The Package Document is the primary source of information about an EPUB Publication, including how to process and display it.

The Package Document is an XML document [XML] consisting of a set of container elements, each dedicated to housing information about a particular aspect of the Publication. These containers effectively centralize metadata for the Publication, detail the individual resources that compose it and provide reading order and other information for rendering the Publication to a User.

The following list summarizes the possible information a Package Document may contain:

  • Publication metadata — mechanisms for including and/or referencing metadata applicable to the entire Publication and particular resources within it.

  • A Publication manifest — identifies (via URI) and describes (via MIME type) the set of resources that collectively compose the Publication.

  • A spine — an ordered sequence of ID references to top-level resources in the manifest from which all other resources in the set can be reached or utilized. The spine defines the default reading order of the publication.

  • Fallback chains — an optional means for publications to define an ordered list of items that can be considered informational equivalents that a Reading System can choose between for rendering content.

  • Bindings — an optional means of associating script-based implementations with custom media types.

› Package Document Content Conformance

› Document Properties

A Package Document must be a well-formed XML Document [XML].

A Package Document must be encoded in UTF-8 or UTF-16 [Unicode].

A Package Document must be valid to the Package Document schema as defined in Appendix A, Package Document Schema and conform to all content conformance constraints expressed in Package Document Definition.

› File Properties

The Package Document filename should use the file extension .opf.

Document fragments included via external general entities [XML] should not use the .opf extension.

› The metadata Element

Element Name

metadata

Usage

Required first child of package.

Attributes

The metadata element has no attributes defined in this specification.

Content Model

In any order: dc:identifier [required], dc:title [required], dc:language [required], meta [optional], link [optional]

The metadata element encapsulates information about the publication as a whole, including metadata that applies to the publication package and to the abstract content contained within the package. Core metadata is expressed by a combination of three required Dublin Core Metadata Element Set [DCMES] properties — title, identifier and language — together with optional properties from DCMI Metadata Terms [DCTERMS] and a set of package and ancillary properties defined in this specification.

Properties from other vocabularies may be added to the metadata section to enhance package and content metadata and to supplement required and optional properties, but their use is optional.

This specification subsets the RDFa 1.1 [RDFa11 Core] profile and prefix definition mechanisms to enable the inclusion and extension of metadata vocabularies for defining publications. These features allow core metadata properties to be added unprefixed, and additionally work to standardize the inclusion of metadata across the publishing spectrum by predefining prefixes for the most commonly-used vocabularies.

All publications must reference the metadata profile on the root package element using the profile attribute. Unlike its RDFa equivalent, the profile attribute must only contain the single canonical URI of the EPUB default profile.

The following example shows how the metadata profile is defined. 

<package … profile="http://www.idpf.org/TODO">
    …
</package>

The addition of the metadata profile allows all properties from the EPUB3 Package Metadata Vocabulary to be referenced without using prefixes. The profile also defines a set of prefixes for use adding other vocabulary properties to EPUB Publications (refer to the profile for more information).

If the metadata profile does not define a prefix for a needed vocabulary, it does not imply that the vocabulary cannot be used. The prefix attribute can also be attached to the package element to define additional prefixes. New prefixes are defined by including whitespace-separated prefix/URI pairings.

The following example shows how to defines a new prefix for a vocabulary. 

<package … prefix="foaf: http://xmlns.com/foaf/spec/">
    <metadata>
        <meta property="dcterms:creator" id="auth">Samuel Johnson</meta>
        <meta property="foaf:title" about="#auth">Dr.</meta>
        …
    </metadata>
</package>

It is not permitted to use the prefix attribute to redeclare a prefix that has already been defined in the metadata profile. The XML Namespaces [XMLNS] method of defining namespace prefixes using the xmlns pseudo-attribute must similary not be used to define vocabulary prefixes.

› The meta Element

Element Name

meta

Usage

As child of the metadata element. Repeatable.

Attributes
›  property [required]

A CURIE [RDFa11 Core] that resolves to a term in one of the vocabularies defined in the metadata profile or package element's prefix attribute.

›  about [context dependent]

Identifies the subject of the property being expressed. The value of the attribute must be a URI [XML] pointing to the resource or element it describes.

The about attribute is optional depending on the type of metadata being expressed. When omitted, the property relates to the EPUB Publication as a whole.

Content Model

Text

The meta element provides a generic means of including package metadata, allowing the expression of primary metadata about the package or content and refinement of that metadata. All meta elements must attach an RDFa property attribute [RDFa11 Core] containing a CURIE that resolves to a property from an RDF vocabulary (it is not required that Reading System be able to resolve the CURIES, however).

If a meta element only includes a property attribute, the metadata is assumed to apply to the package or content as a whole.

The following example shows an instance of package metadata. 

<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
</metadata>

To enahnce metadata properties defined in the DCMES elements or in other meta elements, the RDFa about attribute must be attached. The value of this attribute must be an IDREF [XML] to the xml:id of the element being refined.

The following example shows an instance of ancillary metadata. 

<metadata>
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
    <meta about="#isbn-id" property="scheme">isbn13</meta>
</metadata>

Unless an individual property explicitly specifies a different whitespace normalization algorithm, Reading Systems must trim all leading and trailing whitespace from the meta element values as defined by the XML specification [XML] before further processing them.

› The link Element

Element Name

link

Usage

As a child of metadata. Repeatable.

Attributes
›  href [required]

A URI [RFC2396] to an external metadata resource.

›  rel [required]

A property that describes the nature of the external resource (i.e., its relation to the publication).

Refer to Metadata Record Types for more information.

›  id [optional]

The ID [XML] of this element, which must be unique within the document scope.

Content Model

Empty

The link element is used to associate external metadata resources with the publication. A metadata element may contain one or more link elements, one for each record type.

The href attribute of the link element identifies the location of the resource and the rel attribute defines the type of record (refer to Metadata Record Types for the list of record types).

The following example shows the link element used to associate an ONIX record with the publication. 

<metadata>
    …
    <link rel="onix-record" href="http://example.org/onix/#12389347"/>
    …
</metadata>

The resources identified by the link element's href attribute are considered External Resources, and must therefore not be identified as items in the manifest.

If metadata provided in an external record conflicts with metadata given inline in the Package Document metadata element, then the external metadata should be given precedence.

Reading Systems are not required to dereference the resources identified through the link element.

Examples

The following example shows the link element used to associate two external resources with the publication: an XMP record, and a link to an informational web page. Note that as the foaf:homepage token is not defined by this specification, the metadata extensibility mechanism is employed. 

<package … prefix="foaf: http://xmlns.com/foaf/0.1/">
    <metadata>
        …        
        <link rel="xmp-record" href="http://example.org/xmp/#12389347"/>
        <link rel="foaf:homepage" href="http://example.org/book-info/#12389347" />
        …
    </metadata> 
    …
</package>

› The manifest Element

Element name

manifest

Usage

Required second child of package, following metadata.

Attributes
› id [optional]

The ID [XML] of this element, which must be unique within the document scope.

Content Model

Multiple item [required]

The manifest element provides an exhaustive list of the Publication Resources that constitute the EPUB Publication.

All Publication resources must be referenced from the manifest regardless of their physical location (i.e., regardless of whether they are available in the EPUB Container or remotely). Refer to Publication Resource Locations for media type-specific requirements regarding resource locations.

The manifest is not self-referencing: it must not include item elements referring to the resource(s) that make up the Package Document itself.

Note: 

This specification and its sibling specifications may explicitly define certain resource types to be external. These resources are not Publication Resources and must not be referenced from the manifest. An example of an external resources is a metadata record referenced from the link element.

Note: 

The order of item elements in the manifest is not significant. The presentation sequence of content documents is provided in the spine.

› The item Element

Element Name

item

Usage

As a child of manifest. Repeatable.

Attributes
› id [required]

The ID [XML] of this element, which must be unique within the document scope.

› href [required]

A URI [RFC2396] specifying the location of the Publication Resource described by this item.

› media-type [required]

A MIME string [RFC2046] specifying the media type of the resource described by this item.

› fallback [conditionally required]

An IDREF [XML] that identifies the fallback for a non-Core Media Type.

Refer to Fallback Requirements and Processing Flow for more information.

› property [optional]

A space-separated list of CURIEs [RDFa11 Core] that resolves to one or severals terms in one of the vocabularies defined in the metadata profile or package element's prefix attribute.

Refer to The item Element: Metadata Properties for more information.

› media-overlay [optional]

An IDREF [XML] that identifies the media overlay for the resource described by this item.

Refer to Packaging Media Overlays [MediaOverlays30] for more information.

Content Model

Empty

Each item element in the manifest identifies a Publication Resource by the URI provided in its href attribute. The URI may be absolute or relative. In the case of relative URIs, Reading Systems must use the URI of the Package Document as the base when resolving these to absolute URIs. The resulting absolute URI must be unique within the manifest scope.

The Publication Resource identified by an item must conform to the applicable specification(s) as inferred from the MIME Type provided in the item element media-type attribute. Publication Resources that are Core Media Types must use the MIME Type string designated in EPUB Core Media Types.

All Publication Resources that are not Core Media Types must have an associated fallback as defined in Fallback Requirements and Processing Flow .

All Publication Resources must declare any applicable descriptive metadata properties defined in The item Element: Metadata Properties via the item element's property attribute.

Element name

spine

Usage

Third child of package, following manifest.

Attributes
› id [optional]

The ID [XML] of this element, which must be unique within the document scope.

An IDREF [XML] that identifies the manifest item that represents the superseded NCX.

    <itemref idref="c3-answerkey" linear="no"/>

Content Model

Empty

Each itemref element must (via the idref attribute) reference a manifest item element that represents either an EPUB Content Document, EPUB Navigation Document or a Foreign Content Document.

Consequently, the spine represents an ordered list of a subset of the Publication resources in the manifest, with content items not being referenced being ancillary to those that do appear there.

Note: 

Because of the requirements for fallback, the itemref reference will either a) directly refer to an EPUB Content Document or an EPUB Navigation Document, or b) refer to a Foreign Content Document that includes an EPUB Content Document or an EPUB Navigation Document in its fallback chain.

Note: 

While the EPUB Navigation Document is required in EPUB Publications, it is optional to include it in the spine.

Note: 

The references provided in the spine should resolve to XHTML Content Documents rather than SVG Content Documents, unless a) such use is within a Publication that is packaged together with, and as an alternative non-default representation of, another Publication that represents effectively the same content utilizing XHTML Content Documents or b) the nature of the content being represented via SVG is inherently unsuitable to representation in XHTML.

The itemref element's linear attribute indicates whether a spine item is considered primary (linear='yes') or auxiliary (linear='no').The default value of linear is yes. This may be used to enable Reading Systems to distinguish presentation of "body" content from ancillary content which might be, for example, presented in a popup window, or omitted from an aural rendition.

Reading Systems must provide a means to render a Publication in the order defined by the spine. Reading Systems must as part of this means a) recognize the first primary (linear='yes') item in spine to be the beginning of the main reading order of the publication, and b) render successive primary items in the order given in spine.

› The guide Element [DEPRECATED]

The guide element as defined in [OPF2] is deprecated in EPUB3 in favor of the landmarks feature in the EPUB Navigation Document (refer to The landmarks nav element [ContentDocs30] for further information).

Authors may include the guide element in the Package Document for EPUB 2 Reading System forward compatibility purposes. EPUB3 Reading Systems must ignore the guide element when provided in EPUB3 Publications whose EPUB Navigation Document includes the landmarks feature.

› The bindings Element

Element Name

bindings

Usage

Fourth or fifth child of package, following spine or guide.

Attributes

None.

Content Model

One or several mediaType [required]

The package element may contain at most one bindings element, which contains one or more mediaType elements, each of which associates a custom media type with a "handler" that provides a default script-based implementation for items of that media type when they are referenced from object elements in XHTML Content Documents.

Each mediaType element must uniquely designate, via its required media-type attribute, a unique content type, which must not be a Core Media Type, and must reference, via its required handler attribute, the id in the manifest of the default implementation for this media type, which must be an XHTML Content Document.

Reading Systems that support scripting must utilize the bindings element, if present in a Publication, to handle object element references whose MIME media type is not natively supported by the Reading System. The processing model is to instantiate for such references the designated XHTML Content Document handler (which would typically include scripting) as if it had been referenced from the object element's data attribute, but with URL parameters (which go after '?' in a URL) of

  • src - URI to the resource (value of the data attribute in the object element)

  • type - resource media type (value of the type attribute in the object element)

Any param children of the object element are encoded as URL parameters in the same manner; e.g. for: 

<object data="horse.ogg" media-type="audio/ogg"/>
                        <param name="autoplay" value="false">
                        </object>

a Reading System would instantiate the designated handler Content Document as if its URI query string parameters were: 

src=horse.ogg&type=audio/ogg&autoplay=false

Non-ASCII and non-URL-safe characters (e.g. '=') should be escaped in the usual manner for these parameters (UTF-8 encoding and using %XX escapes).

The following partial example illustrates how bindings can be used to provide a default implementation: 

<manifest>

› Unique Identifier

The Package Document's author is responsible for including a primary identifier that is unique to one and only one particular EPUB Publication. The primary identifier, whether chosen or assigned, must be stored in a dc:identifier element in the Package metadata section and be referenced as the unique identifier in the package element's unique-identifier attibute (refer to dc:identifier for more information).

› Package Identifier

The uniqueness of a publication typically does not change in concert with each revision to the package or its contents, as unique identifiers are intended to have maximal persistence both for referencing and distribution purposes. Each release of a publication normally requires that the new version be uniquely identifiable, however, which results in the contradictory need for reliable unique identifiers that are changeable.

To redress this problem of identifying minor releases without changing the unique identifier, this specification defines the semantics for a Package Identifier, or means of distinguishing and sequentially ordering Publications with the same unique identifer. The Package Identifier is not an actual property in the Package metadata section, but is a value that can be obtained from two required pieces of metadata: the unique identifier and creation or last modification date of the Publication.

When the creation or modification date is taken together with the Publication's unique identifier, the combined value represents a unique identity that can be used to distinguish any particular EPUB instance from another. When a Publication includes both a creation and modification date, the modification date must be used to construct the Identifier.

To ensure that a Package Identifier can be constructed, the following metadata requirements must be satisfied in addition to those specified in metadata:

  • the Publication must include exactly one dcterms:creation property containing the creation date of the Publication. The value of this property must be an ISO 8601 [ISO8601] conformant date of the form TODO: format.

  • the Publication may optionally include one and only one dcterms:modified property containing the last modification date of the Publication. The value of this property must be an ISO 8601 [ISO8601] conformant date of the form TODO: format.

  • if both dcterms:creation and dcterms:modified properties are specified, the dcterms:modified date value must be chronologically after the dcterms:creation date.

The Package Identifier does not supersede the unique identifier, but represents the means by which different versions of the same publication can be identified in distribution channels and by Reading Systems. The chronological order inherent in the required format of the datestamp also places Publications in order without requiring knowledge of the exact identifier that came before.

The Package Identifier consequently allows a set of Publications to be inspected to determine if they represent the same the same version of the same Publication, different versions of a single Publication or any combination of differing and similar Publications.

Note: 

Although it is possible in some instances that the most recent version of a Publication may not be the authoritative version, or that parallel branches of the same Publication may exist, production control problems such as these are outside the scope of this specification.

› Dublin Core Metadata

› Dublin Core Metadata Vocabulary Transition

This specification represents a shift from traditional XML-based grammars to RDFa vocabularies for expressing metadata. Unlike XML grammars, which intrude into host grammars and require specific accommodations to work, RDFa represents a more fluid means of expressing metadata, working within document attributes to add a greater degree of semantic richness.

Although RDFa support for the expression of metadata allows properties from a wide variety of vocabularies to be easily incorporated to express and refine information about an EPUB Publication, it is not expected that all Reading Systems will be able to handle this transition initially. To ensure that older Reading Systems that require the presence of the required DCMES elements do not fail to open new EPUB Publications, this specification represents a transitionary stage to full RDFa support, incorporating both methods of metadata expression.

Although this specification requires the use of the DCMES title, identifier and language elements, their deprecation is expected in a future version. Content authors are consequently encouraged to begin using the new properties from the DCTERMS vocabulary, including the properties that supersede the required DCMES elements. The prefer attribute has been added to each DCMES element to allow authors to designate the new property that replaces the element during this transitionary phase, which also ensures that EPUB Publications created to this specification will continue to function when element support is phased out.

› The DCMES identifier Element

Every metadata section must include at least one identifier element containing an unambiguous identifier for the EPUB Publication. Multiple identifier elements are permitted, but only one can be marked as the primary identifier via the package element's unique-identifier attribute.

The following example shows the unique identifier element for a Publication. 

<package unique-identifier="pub-id">
    <metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
        <dc:identifier id="pub-id">9780101010101</dc:identifier>
    </metadata>
</package>

The [DCTERMS] identifier property will supersede the identifier element in a future version of the specification, but its inclusion is currently optional. Authors should include both versions for forwards compatibility, using the prefer attribute on the identifier element to indicate to Reading Systems that the property should be used in preference.

The following example shows how an identifier element and property should co-exist within a publication. 

<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:identifier id="pub-id" prefer="isbn-id">9780101010101</dc:identifier>
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
</metadata>

This specification imposes no additional restrictions or requirements on identifiers except that they must be at least one character in length.

If the identifier (element or property) conforms to a established system or has been granted by an issuing authority, the scheme employed should be identified by attaching the optional scheme ancillary property to the identifier.

The following example shows how an identifier can be marked as an ISBN. 

<metadata>
    <meta property="dcterms:identifier" id="isbn-id">9780101010101</meta>
    <meta about="#isbn-id" property="scheme">isbn13"</meta>
</metadata>

This specification does not require or endorse the use of any scheme for identifiers, but authors are encouraged to use established identifiers for schemes such as provided by the ONIX and MARC standards. The scheme identifier used should conform to the requirements of the chosen system; this specification imposes no additional restrictions or requirements on scheme identifiers.

› The DCMES title Element

Every metadata section must include at least one title element containing the title for the Publication. Multiple title elements are permitted, but the optional display-seq property should be attached to each to indicate their primacy for display and other rendering purposes. If no means of establishing the primary title for the publication is included, Reading Systems must treat the first title element encountered as the primary title.

The following example shows how to indicate the display order for multiple title elements. 

<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:title id="long-title">On the Origin of Species by Means of Natural Selection</dc:title>
        <meta property="sort-order" about="#long-title">2</meta>
    
    <dc:title id="short-title">On the Origin of Species</dc:title>
        <meta property="sort-order" about="#short-title">1</meta>
    
    <dc:title id="full-title">On the Origin of Species by Means of Natural Selection, 
        or the Preservation of Favoured Races in the Struggle for Life</dc:title>
        <meta property="sort-order" about="#full-title">3</meta>
</metadata>

The [DCTERMS] title property will supersede the title element in a future version of the specification, but its inclusion is currently optional. Authors should include both versions for forwards compatibility, using the prefer attribute on the title element to indicate to Reading Systems that the property should be used in preference.

The following example shows how a title element and property should co-exist within a publication. 

<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:title id="short-title" prefer="dcterms-short-title">On the Origin of Species</dc:title>
        <meta property="dcterms:title" id="short-title">On the Origin of Species</meta>
</metadata>

This specification imposes no additional restrictions or requirements on titles except that they must be at least one character in length.

The file-as property can be attached to a title to include a normalized representation of the title for machine and bibliographic sorting purposes.

The following example shows a title and its normalized form for sorting. 

<metadata xmlns:dc="http://purl.org/dc/elements/1.1/">

<package xml:lang="fr">

› The DCMES language Element

› Fallback Requirements and Processing Flow

› Introduction

EPUB Publications may reference Publication Resources that are not Core Media Types. In order to ensure that all EPUB Publications can be successfully rendered by Reading Systems, alternative "fallback" resources must be provided for each such resource.

Fallbacks are provided either via intrinsic mechanisms defined for certain Publication Resource types, or via a manifest-level fallback mechanism as defined in this section.

› Manifest fallback

Unless one of the exceptions listed in Intrinsic fallback apply, fallbacks must be provided for each Publication Resource that is not a Core Media Type using the fallback attribute on the manifest item element that represents the non-Core Media Type Resource.

The fallback attribute IDREF [XML] value must resolve to another item in the manifest. The ordered list of all ID references that can be reached starting from a given item's fallback attribute is considered to be the "fallback chain" for that item. There must not be any circular- or self references in fallback chains.

A Reading System that does not support the Media Type of a given Publication Resource must traverse the fallback chain until it encounters a supported Publication Resource. The resource identified by the fallback item is then used in place of the unsupported item.

It is legal to provide fallbacks for Publication Resources that are Core Media Types; a Reading System may choose to utilize such fallbacks in order to find the optimal version of a resource to render in a given context. › Intrinsic fallback

› XHTML Content Documents

For every Publication Resource referenced from an XHTML Content Document that is not a Core Media Type, suitable fallback(s) to a resource that is a Core Media Type must be provided as detailed below:

› object

For resources referenced via the object element, legitimate fallback(s) must be provided via that element’s intrinsic replacement capabilities.

Reading Systems must ignore manifest-level fallbacks in resolving such references. (Note that the bindings element could be considered to enable a type of fallback, but not in the sense meant here of ensuring that content can be rendered on all Reading Systems, whether or not they support scripting).

› img

For resources referenced via img elements, the text value of the alt and title attributes provide a legitimate fallback and should be included. Additional manifest-level fallbacks may be provided.

Reading Systems should process manifest-level fallbacks in resolving such references.

› CSS Embedded Fonts

For fonts embedded in Content Documents or EPUB Style Sheets using the @font-face mechanism, the rules of CSS [CSS2.1] must be used to provide an appropriate fallback in case a non-standard or font format is embedded (such fallback may be an OpenType or WOFF embedded font or a reference to a system font).

Reading Systems must ignore manifest-level fallbacks in resolving CSS font references.

› Publication Resource Locations

All Publication Resources must be located in the EPUB Container, with the exceptions listed below.

  • Audio resources, as referenced from Foreign Content Documents, EPUB Content Documents or Media Overlay Documents, may be located in the container or remotely

  • Video resources, as referenced from Foreign Content Documents or EPUB Content Documents, may be located in the container or remotely

Note: 

The inclusion of remote resources in an EPUB Publication is indicated via the online-resources property on the manifest item element.

Authors should prefer locating audio and video resorces in the EPUB Container to allow the user access to the entire presentation regardless of connectivity status.

This schema is normative. In case of conflicts between the specification prose and this schema, the schema shall be considered definitive.

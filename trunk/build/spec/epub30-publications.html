› 1. Overview

› 1.1. Purpose and Scope

This document, EPUB Publications 3.0, defines publication-level semantics and conformance requirements for EPUB3, the third major revision of EPUB, an interchange and delivery format based on XML and Web Standards.

This document is meant to be read and understood in concert with the other documents that make up EPUB3. The EPUB3 Overview [EPUB3Overview], which provides an informative overview of EPUB and a roadmap to the rest of the EPUB3 documents, should be read first.

› 1.2. Relationship to Other Specifications

This specification depends on other sibling specifications, as described in [EPUB3Overview]. Sibling specifications include EPUB Content Documents 3.0 [ContentDocs30], Open Container Format 3.0 [OCF3], and EPUB Media Overlays 3.0 [MediaOverlays30].

This specification defines constructs based on XML [XML], and by default all terminology used in this specification has the same meaning as its use in [XML]. This specification also relies on XML Namespaces [XMLNS], and in this specification the term "XML namespace" (or just "namespace") has the meaning defined in that specification.

EPUB Publications 3.0 is a revision to, and supersedes, Open Package Format 2.0.1 [OPF2].

› 1.3. Terminology

EPUB Content Document (or Content Document)

A document carrying all or parts of the Publication content, as defined in [ContentDocs30].

EPUB Publication (or Publication)

A logical document entity consisting of a set of interrelated resources, as defined by this specification and its sibling specifications.

Extended Content Document Reference

An item in the Package File manifest that either a) refers to an EPUB Content Document, or b) includes an EPUB Content Document in its fallback chain.

External Resource

A resource that is referenced from within an EPUB Publication, but not considered a logical part of the Publication.

Package Document

A document carrying bibliographical and structural metadata about the EPUB Publication.

Publication Resource (or Resource)

A resource that is a logical part of the EPUB Publication. In the absence of this resource, the Publication can not be rendered as intended by the author.

› 2. Introduction

This section is informative

EPUB is an interchange and delivery format for digital publications, based on XML and Web Standards. An EPUB Publication can be thought of as a a reliable packaging of Web content that represents a digital book, magazine, or other type of publication, with a declarative means to navigate through the content, and the ability to be distributed reliably for online and offline consumption.

This specification defines the semantics and conformance requirements for EPUB at the publication level. It defines the formats of the Package and NCX documents, as well as rules for how these documents and other Publication resources are associated to create a conforming EPUB Publication. This specification also defines rules for how EPUB Publications are processed by conforming Reading Systems.

› 3. EPUB Publications

This section describes the high-level content and Reading System conformance requirements for EPUB Publications. The sections following this one describe more specifically the requirements for Package Documents, NCX Documents, and Core Media Types.

› 3.1. Publication Content Conformance

A EPUB Publication minimally consists of the following Publication Resources:

  • Exactly one Package Document, conformant to the requirements defined in Section 4.2, “Package Document Content Conformance”

  • Exactly one NCX Document, conformant to the requirements defined in Section 5.2, “NCX Document Content Conformance”

  • One or more EPUB Content Documents, conformant to the requirements defined for EPUB Content Documents in [ContentDocs30]

  • Zero or more EPUB Style Sheets, conformant to the content conformance requirements defined for EPUB Style Sheets in [ContentDocs30]

  • Zero or more Media Overlay Documents, conformant to the content conformance requirements defined in [MediaOverlays30]

  • Zero or more additional resources, with content conformance requirements inferred from their media type unless explicitly specified differently by this specification or one of its sibling specifications.

An EPUB Publication is instantiated as an EPUB File, i.e. in a ZIP container conformant with the requirements in [OCF3].

› 4. Package Documents

› 4.1. Introduction

This section is informative

The fundamental publication-level construct of an EPUB Publication is the Package Document. The Package Document provides central information about the publication as a whole, and about the individual resources that comprise the publication:

  • Publication metadata - mechanisms for including and/or referencing metadata applicable to the entire Publication and particular resources within it.

  • A Publication manifest, which identifies (via URI) and describes (via MIME type) the set of resources that collectively comprise the Publication.

  • A spine - an ordered sequence of id references to top-level resources in the manifest, from which all other resources in the set can be reached or utilized. The spine defines the default "reading order" of the publication.

  • Navigation - a publication-wide declarative navigation layer.

  • Fallback chains - an optional means for publications to define an ordered list of items that can be considered informational equivalents, that a Reading System can choose between in rendering content

  • Bindings - an optional means of associating script-based implementations with custom media types.

› 4.2. Package Document Content Conformance

A Package Document must be a well-formed XML Document [XML].
A Package Document must be encoded in UTF-8 or UTF-16 [Unicode].
A Package Document must be valid to the Package Document schema as defined in Appendix A, Package Document Schema, and in addition must conform to all content conformance constraints expressed in Section 4.3, “Package Document Definition”.
The Package Document filename should use the file extension .opf. Any document fragments (included via external entities [XML] or XInclude [XInclude]) should not use the .opf extension. TODO check this - are RSs required to support external entites and/or XInclude?

› 4.3.2. The metadata Element

The metadata element contains information about the publication as a whole. This includes metadata that applies to the publication package, as well as the abstract content contained within the package. Information within the metadata element is expressed by a combination of three required Dublin Core Metadata Element Set [DCMES] properties, optional properties from DCMI Metadata Terms [DCTERMS], as well as optional ancillary properties defined in this specification.

The metadata element no longer requires [XMLNS] declarations. This is a change from previous versions, in which the metadata element required declarations to qualify child elements and attributes. This is now achieved via the Metadata Extensibility Mechanism introduced in this specification.

Element Name

metadata

Occurrence

Required first child of package.

Attributes

The metadata element has no attributes defined in this specification.

Content Model

In any order: dc:identifier [required], dc:title [required], dc:language [required], Meta Element [optional], link [optional]

› 4.3.2.1. Dublin Core Metadata Vocabulary Transition

To enhance the ability of metadata to be expressed in a well-understood and semantically-rich form, this specification introduces Dublin Core Metadata Terms [DCTERMS] as the prefered vocabulary for standardized metadata expression. Previous versions of this specification used the Dublin Core Metadata Element Set 1.1 [DCMES] as a vocabulary for expressing standardized metadata. This specification deprecates these legacy Section 4.3.2.2, “Legacy Dublin Core Metadata (DCMES)” elements in favor of elements from [DCTERMS], with the following notable exceptions: dc:identifier, dc:title, dc:language. These three elements are not deprecated in order to maintain the highest level of backwards compatibility while transitioning to [DCTERMS]. These three DCMES elements will be deprecated in a future version of this specification.

› 4.3.2.3. The meta element

The meta element is a generic placeholder to express metadata, or further describe other metadata. This is accomplished by means of an RDFa 1.1 mechanism utilizing the required property attribute, and the optional about attribute.

Element Name

meta

Occurrence

As child of the metadata element. Repeatable.

Attributes
property [required]

Specifies the by name a DCTERMS or ancillary metadata property being expressed.

about [optional unless required]

A reference by id to the subject of the meta element property being expressed. Optional only when a meta element contains a non-empty child text node. When the meta element contains an empty child text-node, the about attribute is required and must contain a valid id of another meta element.

Content Model

Text or Empty.

› 4.3.2.4. The link element

The link element is used to associate external metadata resources with the publication.

Element Name

link

Occurrence

As a child of metadata. Repeatable.

Attributes
href [required]

A valid URL [TODO RFC link] to an external metadata resource.

rel [required]

A token that describes the nature of the external resource (its relation to the publication). See further Section 4.3.2.4.1, “Recognized values for the link element rel attribute”.

id [optional]

An ID [XML] of this element, unique within the document scope.

Content Model

Empty.

The resources identified by the link element href attribute are considered external resources, and must therefore not be identified as items in the manifest.

If metadata provided in an external record conflicts with metadata given inline in the Package Document metadata element, then the external metadata should be given precedence.

Reading Systems are not required to dereference the resources identified through the link element.

› 4.3.2.4.2. The link Element: Examples

The following example shows the link element used to associate an ONIX record with the publication. 

<metadata>
    …
    <link rel="onix-record" href="http://example.org/onix/#12389347"/>
    …
</metadata>

The following example shows the link element used to associate two external resources with the publication: an XMP record, and a link to an informational web page. Note that as the foaf:homepage token is not defined by this specification, the metadata extensibility mechanism is employed. 

<package … prefix="foaf: http://xmlns.com/foaf/0.1/">
    <metadata>
        …        
        <link rel="xmp-record" href="http://example.org/xmp/#12389347"/>
        <link rel="foaf:homepage" href="http://example.org/book-info/#12389347" />
        …
    </metadata> 
    …
</package>

› 4.3.2.6. Metadata extensibility

This specification introduces a mechanism for metadata extensibility that is modeled on the RDFa 1.1 profile mechanism.

By utilizing the package element profile attribute, content authors can incorporate non-prefered metadata vocabularies as needed, and have access to the properties of the non-prefered vocabulary available for use globally in the publication package.

› 4.3.2.7. Scripted Content Documents

In EPUB Publications, Content Documents must not contain any script elements unless the manifest item corresponding to that Content Document contains the property value scripted (thus indicating a Scripted Content Document).Since scripted content may be processed differently, with features such as pagination potentially disabled, content items that set the property value of "scripted" should not be spine items unless absolutely required by a publication (i.e. scripted content should if possible be nested, contained within top-level content via "object" or "iframe").

Reading Systems may render Scripted Content Documents as an interactive, scripted HTML User Agent according to [HTML5] rather than according to the simplified processing model defined for EPUB User Agents in [ContentDocs30], i.e. with scripting enabled. Reading Systems should not enable scripting for any resources other than Scripted Content Documents.

Reading Systems may implement support for Scripted Content Documents in a manner that disables other EPUB capabilities and/or provides a different rendering such as, for example, disabling pagination.

› 4.3.3. The manifest Element

The manifest element, through its item element children, provides an exhaustive list of the Publication Resources that constitute the Publication.

Occurrence

Required second child of package, following metadata.

Attributes
id [optional]

An ID [XML] of this element, unique within the document scope.

Content Model

Multiple item [required]

Any given resource is a Publication Resource if it contains content or instructions that contribute, directly or indirectly, to the rendering of the Publication (e.g. Content Documents, Style Sheets, audio, video, images, embedded fonts, scripts).

All Publication Resources must be referenced from the manifest regardless of their physical location, i.e. regardless of whether they are available in the container or remotely. Refer to Section 4.3.3.2, “Publication Resource Locations” for media type specific requirements regarding resource locations.

This specification and its sibling specifications may explicitly define certain resource types to be external, thereby declaring that the resource is not a Publication Resource. Examples of such external resources include resources referenced from the Package Document metadata link element.

The manifest is not self-referencing: it must not include item elements referring to the resource or resources that make up the Package Document itself.

The order of item elements in the manifest is not significant. The presentation sequence of Content Documents is provided in spine .

› 4.3.3.1. The item element
Element Name

item

Occurrence

As a child of manifest. Repeatable.

Attributes
id [required]

An ID [XML] of this element, unique within the document scope.

href [required]

Specifies the URI of the resource described by this item element.

media-type [required]

Identifies the MIME media type TODO MIME ref of the resource described by this item element.

fallback [conditionally required]

Specifies via IDREF [XML] a fallback for a non-Core Media Type. Refer to Section 4.3.3.1.1, “Item Fallbacks” for details.

property [optional]

Specifies metadata for the given item. Refer to Section 4.3.3.1.2, “Recognized values for the item element property attribute”.

media-overlay [optional]

Defines via IDREF [XML] the media overlay for the resource described by this item element. Refer to Section 3.3 of [MediaOverlays30] for details.

The URI provided in the item element href attribute may be relative, and is in this case resolved to an absolute URI using the URI of the Package Document as the base. The resulting absolute URI must be unique within the manifest scope.

The resource identified by an item must conform to the applicable specification(s) as inferred from the MIME Type provided in the item element media-type attribute.

› 4.3.3.1.1. Item Fallbacks

Each item in the manifest whose media-type value is not one of the EPUB3 Core Media Types must use the fallback attribute to specify a fallback chain that includes at least one item whose media-type is an EPUB3 Core Media Type.

The fallback attribute IDREF [XML] value must resolve to another item in the manifest. The ordered list of all ID references that can be reached starting from a given item's fallback attribute is considered to be the "fallback chain" for that item. There must not be any circular references in fallback chains.

The general rule for processing manifest-level fallbacks is that a Reading System that does not support the Media Type of a given item must traverse the fallback chain until it encounters a supported Media Type. The resource identified by the fallback item is then used in place of the unsupported item. It is legal to have a fallback attribute on an item of a Core Media Type and a Reading System may choose to "fall over" such references in order to find the optimal version of a resource to render in a given context. TODO Fallbacks for NCX, and possibly other core media types, are not legal. Determine which they are and add here.

However, Reading Systems process fallbacks differently depending on the context where a reference to a resources occurs in Content Documents, in conjunction with the intrinstic fallback mechanisms defined in [HTML5] and [SVG] . Section 6.2, “Fallback Requirements and Processing Flow ” defines the overall processing flow for handling fallbacks.

› 4.3.6. The bindings element

Occurrence

Fourth or fifth child of package, following spine or guide.

Attributes

None.

Content Model

One or several item [required]

The package element may contain at most one bindings element, which contains one or more mediaType elements, each of which associates a custom media type with a "handler" that provides a default script-based implementation for items of that media type when they are referenced from object elements in XHTML Content Documents.

Each mediaType element must uniquely designate, via its required media-type attribute, a unique content type, which must not be a Core Media Type, and must reference, via its required handler attribute, the id in the manifest of the default implementation for this media type, which must be an XHTML Content Document.

Reading Systems that support scripting must utilize the bindings element, if present in a Publication, to handle object element references whose MIME media type is not natively supported by the Reading System. The processing model is to instantiate for such references the designated XHTML Content Document handler (which would typically include scripting) as if it had been referenced from the object element's data attribute, but with URL parameters (which go after '?' in a URL) of

  • src - URI to the resource (value of the data attribute in the object element)

  • type - resource media type (value of the type attribute in the object element)

Any param children of the object element are encoded as URL parameters in the same manner; e.g. for: 

<object data="horse.ogg" media-type="audio/ogg"/>
<param name="autoplay" value="false">
</object>

a Reading System would instantiate the designated handler Content Document as if its URI query string parameters were: 

src=horse.ogg&type=audio/ogg&autoplay=false

Non-ASCII and non-URL-safe characters (e.g. '=') should be escaped in the usual manner for these parameters (UTF-8 encoding and using %XX escapes).

  <item id="slideshow" href="slideshow.xml" media-type="application/x-demo-slideshow"/>
 </manifest>
 <bindings>
   <mediaType media-type="application/x-demo-slideshow" handler="impl"/>
 </bindings>

in content.xhtml:

...

<!-- if scripting is not available, just show them inline -->

› 5. NCX Documents

› 5.1. Introduction

This section is informative

In order to increase efficacy and richness of navigation, EPUB Publications include a "Navigation Center eXtended," the NCX. The NCX provides a means for the author to provide a multi-purpose declarative global navigation layer that is independent of the content it references.

The NCX was originally standardized by the DAISY Consortium, as defined in Section 8 of [Z3986-2005] . The EPUB specification defines its own schema (XML grammar) for the NCX content type, as some multimedia and metadata components from the DAISY specification are not included.

› 5.1.1. Key NCX Components

The following are the key components of the NCX document:

navMap

The navMap exposes the hierarchical structure of the EPUB Publication. The provided structure is global, i.e. it transcends the boundaries of each individual EPUB Content Document of [ContentDocs30] in the spine.

In terms of the number of links provided, the navMap is often more extensive than a traditional print table of contents.

navList

A navList is a container for non-hierarchical navigational information that is secondary to the navMap.

Multiple navLists can be provided, either representing constructs with homogenous semantics (such as list of illustrations, or all quotes of a certain author in an anthology) or with heterogenous semantics (such as the the landmarks navList).

pageList

The pageList is a special-purpose navList that allows representing the location of page boundaries present in a print source. Metadata for bibliographical identification of the print source can be defined via the navMeta element (and its meta children).

› 5.1.2. NCX Metadata

The NCX provides two mechanisms for expressing metadata:

Document metadata

The NCX head element allows the expression of metadata concerning the NCX Document as a whole. This version of this specification does not require head metadata to be provided.

Link target metadata

The epub:type attribute, as occurring on the navList, navPoint and navTarget elements, provides machine-readable metadata about the structural semantics of link target(s). Converserly, the navLabel element provides human-readable information.

Additionally, the navMeta element can be used to define metadata information (using the meta element) for the navList, navMap and pageList containers, as well as for the navPoint, navTarget and pageTarget elements.

Mappings to metadata vocabularies used when expressing document and link target metadata are created using the profile and prefix attributes on the ncx root element . The EPUB Structural Semantics Vocabulary is the default metadata vocabulary of the NCX.

› 5.2. NCX Document Content Conformance

An NCX Document must be a well-formed XML Document [XML].
An NCX Document must be encoded in UTF-8 or UTF-16 [Unicode].
An NCX Document must be valid to the NCX Document schema as defined in Appendix B, NCX Document Schema, and in addition must conform to all content conformance constraints expressed in Section 5.4, “NCX Document Definition”.
The NCX Document filename should use the file extension .ncx. Any document fragments (included via XML entities [XML] or XInclude) [XInclude] should not use the .ncx extension.

› 5.3. NCX Reading System Conformance

EPUB Reading Systems must provide means to reach parts of a publication based on the information available within the NCX, by exposing the hierarchical navigation structure and the navigation lists specified by navMap, navList and pageList.
EPUB Reading Systems must use human-readable descriptions encoded within the navLabel element as authoritative information for rendering the hierarchical navigation structure and the navigation lists.

The NCX pageTarget element specifies a pointer to content and briefly describes the target.

› 5.4.6.1. The landmarks navList

› 5.4.9. The navLabel Element

The NCX navLabel element briefly describes the target in the EPUB Content Document of [ContentDocs30] . When applied to navPoints, it generally contains the heading text of the referenced section of the document.

There can be multiple navLabel elements, for example to provide information in various languages. Several navLabel elements with the same language (see xml:lang definition below) are forbidden. Since a navLabel element without the optional xml:lang attribute inherits the language specified for the ncx top-level element, only one navLabel element without the xml:lang attribute is allowed. Reading systems should first attempt to find a navLabel matching the language specified by user preferences, secondly matching the language specified by the package's dc:language metadata element, thirdly matching the language specified by the xml:lang attribute on the ncx element, and lastly by defaulting to the first available navLabel element (in XML document order).

Specifies the language used in the contents and attribute values of the carrying element and its descendants, as defined in section 2.12 Language Identification of [XML].

Inherent directionality specified using [Unicode] takes precedence over this attribute.

› 6.2. Fallback Requirements and Processing Flow

This section defines in detail the requirements for including fallback content in a Publication, and the process by which Reading Systems must process fallback information. The rules for fallback processing depend on both the type of item, and how is referenced by a Content Document or other resource.

This specification defines a set of Core Media Types that all conforming Reading Systems must support in addition to Content Documents (e.g. PNG, SVG, OpenType fonts). Publications may optionally reference items of additional media types. In order to ensure that all conforming Publications can be successfully rendered by all conforming Reading Systems, alternative "fallback" items must be provided for each such item. Fallbacks may be provided either via inherent mechanisms defined for Content Documents, or via the manifest-level fallback mechanism defined in this specification.

Any Reading System that supports video playback must support video files that utilize the MPEG-4 container with H.264/MPEG-4 AVC baseline profile format (H.264) video codec and MPEG-4 Advanced Audio Coding - Low Complexity (AAC-LC) audio codec, and this format is a Core Media Type.

Any Reading System that supports audio playback must support MPEG-1 Layer 3 (MP3) audio, and this format is a Core Media Type. Any Reading System that supports video playback must also support standalone AAC-LC audio playback, but AAC-LC is NOT a Core Media Type.

For every item that is not a Core Media Type, suitable fallback(s) of a Core Media Type must be provided via one of these mechanisms, as detailed below:

For resources referenced in Content Documents via object elements, legitimate fallback(s) must be provided via that element’s inherent replacement capabilities.

Reading Systems must ignore manifest-level fallbacks in resolving such references (the bindings element described below could be considered to enable a type of fallback, but not in the sense meant here of ensuring that content can be rendered on all Reading Systems, whether or not they support scripting).

For resources referenced in Content Documents via img elements, the text value of the alt or title attributes provide a legitimate fallback and should be included. Additional manifest-level fallbacks may be provided.

Reading Systems should process manifest-level fallbacks in resolving such references.

For fonts embedded in Content Documents or EPUB Style Sheets using the @font-face mechanism, the basic rules of CSS must be used to provide an appropriate fallback in case a non-standard font format is embedded (such fallba\ck may be an an OpenType or WOFF embedded font or a reference to a system font).

Reading Systems must support both OpenType and WOFF embededed fonts as well as the font obfuscation mechanism defined in [OCF].

SVG fonts may be referenced from CSS using the @font-face mechanism but Reading System support for SVG fonts is optional. I.e. when SVG fonts are used a fallback is still required as described above.

Reading Systems must ignore manifest-level fallbacks in resolving CSS font references.

For resources referenced in Content Documents via iframe elements, only XHTML Content Documents may be referenced; fallback processing is inapplicable.

For all other resource referencing situations in EPUB Publications, a legitimate fallback must be provided via the manifest fallback mechanism.

Reading Systems must process manifest-level fallbacks in resolving such resource references.
